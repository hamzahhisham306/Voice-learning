01

The big picture — what is Converso?

Converso is a “real-time AI teaching platform.” A user creates an AI companion (a tutor) by choosing a name, subject, topic, voice, and speaking style. They then start a live voice conversation with that tutor — talking out loud, hearing spoken replies, and watching a live transcript. Completed sessions are recorded, and favourite companions can be bookmarked.

Three external services each own one responsibility, and the app glues them together:

Identity

Clerk

Who the user is. Sign in / sign up, session cookies, and subscription billing (plan limits).

Data

Supabase

What the user has. A Postgres database storing companions, session history, and bookmarks.

Voice

Vapi

The live conversation. Turns microphone audio into a spoken AI tutor in real time.

Everything meets in two places: the Next.js server (for data, guarded by Clerk) and the browser (for the live voice call).

The single most important idea: the three services never talk to each other directly. The Next.js app is the hub. Clerk proves identity, that identity unlocks Supabase, and the browser drives Vapi — then reports the result back to the server, which writes to Supabase.

Mental model

03

Clerk: Sign In & Sign Up

Clerk is a drop-in authentication and user-management service. Instead of building your own login, password hashing, OAuth, sessions, and account UI, you render Clerk’s components and it handles all of it — including “Continue with Google/GitHub,” email codes, passkeys, and secure session cookies.

The pieces in this app

  • ClerkProvider wraps the whole app in app/layout.tsx, making auth state available everywhere.
  • <SignInButton> / <SignUpButton> in the navbar open Clerk’s hosted flow. <SignedIn> / <SignedOut> conditionally show UI, and <UserButton> is the profile avatar menu.
  • Catch-all routes app/sign-in/[[...sign-in]]/page.tsx and app/sign-up/[[...sign-up]]/page.tsx host the full sign-in/sign-up pages. The [[...x]] is an optional catch-all so Clerk can own every sub-path (e.g. verification, SSO callback) under one route.
  • middleware.ts runs clerkMiddleware() on every request, reading the session cookie so the server knows who you are.
components/Navbar.tsx — auth controls
// Signed-out users see Sign In + Sign Up; signed-in users see their avatar
<SignedOut>
  <SignInButton><button className="btn-signin">Sign In</button></SignInButton>
  <SignUpButton><button className="btn-primary">Sign Up</button></SignUpButton>
</SignedOut>
<SignedIn>
  <UserButton />
</SignedIn>

What actually happens (the auth flow)

Sign Up creates a brand-new user record in Clerk. Sign In authenticates an existing user. Both end the same way: Clerk sets a session cookie (a signed JWT) in the browser. From then on, every request carries that cookie, and the server can call auth() to get the user’s ID.

  participant B as Browser (Next.js)

  U->>B: Click "Sign In" / "Sign Up"
  B->>C: Open Clerk UI (hosted flow)
  U->>C: Email + code / Google / passkey
  C-->>B: Set session cookie (signed JWT)
  Note over B,C: User is now authenticated
  U->>B: Visit a protected page

  C-->>MW: Valid — userId = the "sub" claim
  MW-->>B: Render page; auth() / currentUser() now work
Sign in / sign up both result in a session cookie the middleware validates on every request.

Reading the user on the server

Two helpers from @clerk/nextjs/server are used throughout:

  • auth() — lightweight; returns { userId, has, getToken }. It is async — always await it.
  • currentUser() — the full user object (name, email, image), used on the profile/lesson pages.
app/companions/[id]/page.tsx
const user = await currentUser();
if (!user) redirect('/sign-in'); // guard: bounce anonymous users

Clerk also does billing

The app/subscription/page.tsx page is literally just <PricingTable /> — a Clerk component that renders subscription plans. The app then checks entitlements with has() to enforce limits (see the “create companion” flow later):

lib/actions/companion.actions.ts — newCompanionPermissions()
const { userId, has } = await auth();
if (has({ plan: 'pro' })) return true;          // unlimited
else if (has({ feature: "3_companion_limit" })) limit = 3;
else if (has({ feature: "10_companion_limit" })) limit = 10;

Supabase — the database layer

Supabase is an open-source “Firebase alternative” built on PostgreSQL. You get a real SQL database plus a set of tools around it. Its main features:

  • Postgres database — full relational SQL, foreign keys, indexes, constraints.
  • Auto-generated REST API (PostgREST) — every table is instantly queryable via the JS client (supabase.from('companions').select()). No backend endpoints to write.
  • Row-Level Security (RLS) — per-row access rules enforced inside the database, so users only see their own data.
  • Auth, Realtime, Storage, Edge Functions — extra features (this app uses only the database + RLS; Clerk handles auth instead).

Typical use cases: the backend for web/mobile apps, dashboards, SaaS products — anywhere you want a Postgres database with an instant API and built-in authorization, without running your own server.

How this app uses Supabase

Three tables (all IDs are uuid; the Clerk user id is stored as text):

TableColumnsPurpose
companionsid, name, subject, topic, voice, style, duration, author, created_atEach AI tutor. author = the Clerk id of who created it.
session_historyid, companion_id→companions, user_id, created_atOne row per completed voice lesson.
bookmarksid, companion_id→companions, user_id, created_atA user’s saved companions (unique per user+companion).

The client is built with the Clerk token — the crucial detail

This is the heart of the Clerk↔Supabase integration. Every time the app needs the database, it builds a fresh Supabase client and hands it a function that returns the current Clerk session token:

lib/supabase.ts
import { createClient } from "@supabase/supabase-js";
import { auth } from "@clerk/nextjs/server";

export const createSupabaseClient = () => {
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
    {
      async accessToken() {
        // Give Supabase the logged-in user's Clerk JWT
        return (await auth()).getToken();
      },
    }
  );
};

Supabase receives that JWT, verifies it against Clerk’s public keys (configured as a Third-Party Auth provider), and then RLS policies use the token’s sub claim (the Clerk user id) to filter rows:

RLS policy (SQL)
-- a user can only read their own bookmarks
create policy "bookmarks_select_own" on public.bookmarks
  for select to authenticated
  using ((auth.jwt() ->> 'sub') = user_id);
Why not auth.uid()?

Supabase’s built-in auth.uid() expects a UUID. Clerk user ids look like user_2abc… (not a UUID), so the policies use auth.jwt() ->> 'sub' to read the raw Clerk id as text.

05

Vapi Voice — the real-time tutor

Vapi is a voice-AI platform. It orchestrates the three moving parts of a spoken conversation so you don’t have to wire them yourself:

  1. Speech-to-text (transcriber) — here Deepgram (nova-3) turns the user’s microphone audio into text.
  2. The “brain” (LLM) — OpenAI GPT-4 receives that text plus a tutor system prompt and produces a reply.
  3. Text-to-speech (voice) — ElevenLabs (11labs) speaks the reply in the chosen voice.

Crucially, in this app Vapi runs entirely in the browser via @vapi-ai/web. There is no server involved in the live call.

Configuring the assistant

configureAssistant(voice, style) in lib/utils.ts builds the Vapi config object. Note the {{topic}}, {{subject}}, {{style}} placeholders — these are filled in at call time (see below):

lib/utils.ts — configureAssistant() (excerpt)
const voiceId = voices[voice][style] || "sarah";   // pick an ElevenLabs voice
return {
  transcriber: { provider: "deepgram", model: "nova-3", language: "en" },
  voice:       { provider: "11labs", voiceId, stability: 0.4, /* … */ },
  model: {
    provider: "openai", model: "gpt-4",
    messages: [{ role: "system", content: `You are a tutor teaching {{ subject }}…
      Stick to the topic {{ topic }}. Keep your style {{ style }}. …` }],
  },
};

Driving the call: components/CompanionComponent.tsx

This client component is the entire live-session UI. It (1) starts/stops the call, (2) subscribes to Vapi events, and (3) renders the transcript + animations. A local CallStatus state machine (INACTIVE → CONNECTING → ACTIVE → FINISHED) drives the button and visuals.

components/CompanionComponent.tsx — starting the call
const handleCall = async () => {
  setCallStatus(CallStatus.CONNECTING);
  const assistantOverrides = {
    variableValues: { subject, topic, style },  // fills {{subject}} etc.
    clientMessages: ["transcript"],           // send transcripts to the browser
    serverMessages: [],                          // no server webhooks
  };
  vapi.start(configureAssistant(voice, style), assistantOverrides);
};
components/CompanionComponent.tsx — listening to Vapi events
vapi.on('call-start', () => setCallStatus(ACTIVE));
vapi.on('call-end',   () => { setCallStatus(FINISHED); addToSessionHistory(companionId); });
vapi.on('message',    (m) => {          // live transcript
  if (m.type === 'transcript' && m.transcriptType === 'final')
    setMessages(prev => [{ role: m.role, content: m.transcript }, ...prev]);
});
vapi.on('speech-start', () => setIsSpeaking(true));   // play soundwave animation
vapi.on('speech-end',   () => setIsSpeaking(false));
Note

The live call needs the browser env var NEXT_PUBLIC_VAPI_WEB_TOKEN (see lib/vapi.sdk.ts). If it’s not set, the page still renders but vapi.start() can’t connect. This is separate from Clerk/Supabase.

How Clerk, Vapi & Supabase communicate

They don’t connect in a triangle — they connect through the app. Here’s the same picture as responsibilities and the “glue” between each pair:

flowchart TB
  APP(["Converso app
(Next.js + Browser)"])
  CLERK["🔐 Clerk"]
  SUPA["🗄️ Supabase"]
  VAPI["🎙️ Vapi"]
  APP -- "auth() · cookies · has()" --- CLERK
  APP -- "supabase.from(...)
server actions" --- SUPA
  APP -- "vapi.start/stop · events" --- VAPI
  CLERK == "issues JWT that
Supabase verifies (Third-Party Auth)" ==> SUPA
The only direct link between two services is Clerk → Supabase, and it’s indirect: Clerk mints a token, Supabase is configured to trust it.

Clerk ↔ Supabase (identity unlocks data)

Clerk mints a session JWT; createSupabaseClient() attaches it to every query; Supabase verifies it via the Clerk domain registered under Third-Party Auth, then RLS filters rows by sub. No webhooks — it’s request-time token verification.

App ↔ Vapi (the browser drives the call)

The browser opens a WebRTC call to Vapi and listens for events. Vapi never touches Supabase or Clerk. When the call ends, the browser calls a server action to persist the session.

App ↔ Clerk (identity + billing)

clerkMiddleware() validates the cookie on each request; auth()/currentUser() read the user; has() checks the subscription plan.

About webhooks

You asked about webhooks. This app doesn’t use any. Vapi is configured with serverMessages: [] (no server callbacks), and Clerk’s plan checks happen at request time via has(). Everything is either a server action (browser → Next server → Supabase) or a request-time token check. Webhooks could be added later (e.g. Vapi serverMessages to log full transcripts server-side, or Clerk webhooks to sync users) — but they aren’t part of the current design.

Vapi ↔ Supabase: how data flows

Vapi and Supabase never speak directly. The browser is the bridge. During a call, transcripts live only in React state (they’re shown, not saved). When the call ends, the browser fires a server action that writes one row to session_history.

sequenceDiagram
  actor U as User
  participant CC as CompanionComponent (browser)
  participant V as Vapi (cloud)
  participant SA as addToSessionHistory (server action)
  participant S as Supabase
  U->>CC: Click "Start Session"
  CC->>V: vapi.start(assistant, { variableValues })
  V-->>CC: 'call-start' → status ACTIVE
  loop during the lesson
    U->>V: speaks (mic audio)
    V-->>CC: 'message' transcript → append to React state
    V-->>CC: 'speech-start' / 'speech-end' → animate soundwave
  end
  U->>CC: Click "End Session"
  CC->>V: vapi.stop()
  V-->>CC: 'call-end'
  CC->>SA: addToSessionHistory(companionId)
  SA->>SA: userId = (await auth()).userId
  SA->>S: insert session_history { companion_id, user_id }
  S-->>SA: ok (RLS: sub === user_id)
The only Vapi data that reaches Supabase is “this user completed a session with this companion.”
lib/actions/companion.actions.ts — addToSessionHistory()
export const addToSessionHistory = async (companionId) => {
  const { userId } = await auth();               // Clerk id
  const supabase = createSupabaseClient();        // carries the Clerk JWT
  await supabase.from('session_history').insert({
    companion_id: companionId,
    user_id: userId,
  });
};

That single record powers “Lessons completed,” “Recent Sessions,” and the home page’s “Recently completed sessions” — all read back with getRecentSessions() / getUserSessions(), which use a Postgres foreign-key embed to pull the full companion for each session:

getUserSessions() — reading back with an embed
supabase
  .from('session_history')
  .select(`companions:companion_id (*)`)  // follow the FK → full companion
  .eq('user_id', userId)
  .order('created_at', { ascending: false });
Why the foreign key mattered

That companions:companion_id (*) embed only works because session_history.companion_id has a foreign key to companions(id). The same is true for bookmarks. Missing that FK is why /my-journey crashed until the tables were created correctly.

Example B — Running a voice lesson

  1. User opens /companions/{id}. The server loads the companion and the Clerk user, then renders CompanionComponent with props (subject, topic, voice, style, userName…).
  2. User clicks Start Session → vapi.start(configureAssistant(voice,style), { variableValues }). Status → CONNECTING.
  3. Vapi fires call-start → status ACTIVE. The tutor greets the user; Deepgram/GPT-4/11Labs run in a loop.
  4. message events stream in → each final transcript is prepended to the on-screen conversation. speech-start/end toggle the soundwave animation.
  5. User clicks End Session → vapi.stop() → call-end → addToSessionHistory(companionId) writes to Supabase.

Example C — Bookmarking (and why the page refreshes)

Clicking the bookmark on a card calls addBookmark(companionId, pathname). It inserts a row, then calls revalidatePath(pathname) so Next.js re-fetches the server component and the UI reflects the new state. RLS guarantees the row’s user_id must equal the caller’s Clerk id.

Key concepts & dependencies

Server Components vs. Client Components

By default, App Router files are Server Components — they run on the server, can be async, and can hit Supabase/Clerk directly (great for data-loading pages). A file that needs browser APIs, state, or event handlers starts with 'use client' (e.g. CompanionComponent, CompanionForm, SearchInput). Vapi must be a client component because it uses the microphone.

Server Actions

Files marked 'use server' expose functions the client can call like a normal await, but they execute on the server. This is how the browser safely triggers database writes (e.g. addToSessionHistory, createCompanion) without exposing DB credentials.

Row-Level Security (RLS)

Authorization lives in Postgres, not just in app code. Even if a query forgot a .eq('user_id', …), RLS would still restrict rows to the current user. The policies key off the Clerk sub claim.

One dashboard step that isn’t in code

For Supabase to trust Clerk’s tokens, Clerk is registered as a Third-Party Auth provider in the Supabase dashboard (using the Clerk domain, e.g. sure-piranha-43.clerk.accounts.dev), and the Supabase integration is enabled in the Clerk dashboard. That configuration is what makes the whole Clerk→Supabase chain work — nothing in the repo shows it.
