01

الصورة الكاملة — ما هو كونفيرسو؟

كونفيرسو (Converso) هو «منصّة تعليم بالذكاء الاصطناعي في الوقت الفعلي». ينشئ المستخدم رفيقًا (companion) — أي معلّمًا ذكيًّا — باختيار اسم وموضوع ومحور ونبرة صوت وأسلوب حديث. ثم يبدأ محادثة صوتيّة حيّة مع هذا المعلّم: يتحدّث بصوته، ويسمع ردودًا منطوقة، ويشاهد النص المكتوب مباشرةً. تُسجَّل الجلسات المكتملة، ويمكن حفظ الرفقاء المفضّلين في المفضّلة.

ثلاث خدمات خارجيّة، كلٌّ منها مسؤولة عن جانب واحد، والتطبيق هو الذي يربطها معًا:

الهويّة

Clerk

مَن هو المستخدم. تسجيل الدخول والتسجيل، وملفّات الجلسة (cookies)، وفوترة الاشتراكات (حدود الخطط).

البيانات

Supabase

ما الذي يملكه المستخدم. قاعدة بيانات Postgres تُخزّن الرفقاء وسجلّ الجلسات والمفضّلة.

الصوت

Vapi

المحادثة الحيّة. تُحوّل صوت الميكروفون إلى معلّم ذكيّ يتكلّم في الوقت الفعلي.

أهمّ فكرة على الإطلاق: الخدمات الثلاث لا تتحدّث مع بعضها مباشرةً. تطبيق Next.js هو المركز (hub). يُثبِت Clerk الهويّة، وتلك الهويّة تفتح الوصول إلى Supabase، والمتصفّح يُشغّل Vapi — ثم يُبلِّغ النتيجة إلى الخادم، الذي يكتب بدوره في Supabase.

03

Clerk: تسجيل الدخول والتسجيل

Clerk خدمة جاهزة للمصادقة وإدارة المستخدمين. بدلًا من بناء نظام تسجيل دخول وتشفير كلمات مرور وجلسات وواجهات حسابات من الصفر، تعرض مكوّنات Clerk فيتكفّل هو بكلّ ذلك — بما في ذلك «المتابعة عبر Google/GitHub»، ورموز البريد الإلكتروني، ومفاتيح المرور (passkeys)، وملفّات جلسة آمنة.

الأجزاء المستخدَمة في هذا التطبيق

  • ClerkProvider يُغلّف التطبيق كلّه في app/layout.tsx، فيجعل حالة المصادقة متاحة في كلّ مكان.
  • <SignInButton> و<SignUpButton> في شريط التنقّل يفتحان تدفّق Clerk المستضاف. و<SignedIn> و<SignedOut> يعرضان الواجهة بشكل مشروط، و<UserButton> هو قائمة صورة الحساب.
  • المسارات الشاملة (catch-all) app/sign-in/[[...sign-in]]/page.tsx وapp/sign-up/[[...sign-up]]/page.tsx تستضيف صفحات تسجيل الدخول والتسجيل الكاملة. الصيغة [[...x]] هي «مسار شامل اختياري» تتيح لـ Clerk امتلاك كلّ المسارات الفرعيّة (كالتحقّق أو ردّ نداء SSO) تحت مسار واحد.
  • middleware.ts يُشغّل clerkMiddleware() في كلّ طلب، فيقرأ ملفّ الجلسة ليعرف الخادم من أنت.
components/Navbar.tsx — أزرار المصادقة
// غير المسجّلين يرون Sign In + Sign Up؛ المسجّلون يرون صورة حسابهم

ماذا يحدث فعليًّا (تدفّق المصادقة)

التسجيل (Sign Up) يُنشئ سجلّ مستخدم جديدًا تمامًا في Clerk. أمّا تسجيل الدخول (Sign In) فيُصادِق على مستخدم موجود. وكلاهما ينتهي بالطريقة نفسها: يضع Clerk ملفّ جلسة (cookie) — وهو رمز JWT موقَّع — في المتصفّح. ومن تلك اللحظة، يحمل كلّ طلب هذا الملفّ، ويستطيع الخادم استدعاء auth() للحصول على معرّف المستخدم.

تسجيل الدخول والتسجيل كلاهما ينتهي بملفّ جلسة يتحقّق منه الوسيط (middleware) في كلّ طلب.

قراءة المستخدم على الخادم

تُستخدَم دالّتان مساعدتان من @clerk/nextjs/server في كلّ أنحاء التطبيق:

  • auth() — خفيفة؛ تُعيد { userId, has, getToken }. وهي غير متزامنة (async) — انتظرها دائمًا بـ await.
  • currentUser() — كائن المستخدم الكامل (الاسم، البريد، الصورة)، يُستخدَم في صفحتَي الملفّ الشخصي والدرس.
app/companions/[id]/page.tsx
const user = await currentUser();
if (!user) redirect('/sign-in'); // حماية: إعادة الزوّار غير المسجّلين

Clerk يتولّى الفوترة أيضًا

صفحة app/subscription/page.tsx هي حرفيًّا مجرّد <PricingTable /> — وهو مكوّن من Clerk يعرض خطط الاشتراك. ثم يتحقّق التطبيق من الصلاحيّات عبر has() لفرض الحدود (انظر مسار «إنشاء رفيق» لاحقًا):

lib/actions/companion.actions.ts — newCompanionPermissions()
const { userId, has } = await auth();
if (has({ plan: 'pro' })) return true;          // بلا حدود
else if (has({ feature: "3_companion_limit" })) limit = 3;
else if (has({ feature: "10_companion_limit" })) limit = 10;

04

Supabase — طبقة قاعدة البيانات

Supabase بديل مفتوح المصدر لـ Firebase، مبنيّ على PostgreSQL. تحصل على قاعدة بيانات SQL حقيقيّة مع مجموعة أدوات حولها. أبرز مزاياها:

  • قاعدة بيانات Postgres — SQL علائقيّ كامل، مع مفاتيح أجنبيّة وفهارس وقيود.
  • واجهة REST مولّدة تلقائيًّا (PostgREST) — كلّ جدول قابل للاستعلام فورًا عبر مكتبة JS (supabase.from('companions').select()). لا حاجة لكتابة نقاط نهاية خلفيّة.
  • أمان مستوى الصف (RLS) — قواعد وصول لكلّ صف تُفرَض داخل قاعدة البيانات نفسها، فلا يرى المستخدم إلّا بياناته.
  • مصادقة وتحديث لحظيّ وتخزين ودوالّ حافّة — ميزات إضافيّة (هذا التطبيق يستخدم قاعدة البيانات + RLS فقط؛ فالمصادقة يتولّاها Clerk).

حالات الاستخدام المعتادة: الطبقة الخلفيّة لتطبيقات الويب والجوّال، ولوحات المعلومات، ومنتجات SaaS — أي مكان تريد فيه قاعدة Postgres مع واجهة فوريّة وتفويض مدمج، دون تشغيل خادمك الخاص.

كيف يستخدم هذا التطبيق Supabase

ثلاثة جداول (كلّ المعرّفات من نوع uuid؛ ومعرّف مستخدم Clerk مُخزَّن كنصّ text):

الجدولالأعمدةالغرض
companionsid, name, subject, topic, voice, style, duration, author, created_atكلّ معلّم ذكيّ. author = معرّف Clerk لمن أنشأه.
session_historyid, companion_id→companions, user_id, created_atصفّ واحد لكلّ درس صوتيّ مكتمل.
bookmarksid, companion_id→companions, user_id, created_atالرفقاء المحفوظون للمستخدم (فريد لكلّ مستخدم+رفيق).

يُبنى العميل مع رمز Clerk — التفصيلة الحاسمة

هذا هو جوهر تكامل Clerk↔Supabase. في كلّ مرّة يحتاج التطبيق فيها قاعدة البيانات، يبني عميل Supabase جديدًا ويُمرّر له دالّة تُعيد رمز جلسة Clerk الحاليّ:

lib/supabase.ts
import { createClient } from "@supabase/supabase-js";
import { auth } from "@clerk/nextjs/server";

export const createSupabaseClient = () => {
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
    {
      async accessToken() {
        // أعطِ Supabase رمز JWT الخاص بمستخدم Clerk المسجّل
        return (await auth()).getToken();
      },
    }
  );
};

يستقبل Supabase ذلك الرمز، ويتحقّق منه مقابل مفاتيح Clerk العموميّة (المُعرَّفة كمزوّد مصادقة طرف ثالث)، ثم تستخدم سياسات RLS المطالبة sub في الرمز (وهي معرّف مستخدم Clerk) لتصفية الصفوف:

سياسة RLS (SQL)
-- يستطيع المستخدم قراءة مفضّلته هو فقط
create policy "bookmarks_select_own" on public.bookmarks
  for select to authenticated
  using ((auth.jwt() ->> 'sub') = user_id);
لماذا لا نستخدم auth.uid()؟

دالّة auth.uid() المدمجة في Supabase تتوقّع قيمة UUID. لكنّ معرّفات مستخدمي Clerk تبدو هكذا user_2abc… (وليست UUID)، لذا تستخدم السياسات auth.jwt() ->> 'sub' لقراءة معرّف Clerk الخام كنصّ.

05

Vapi الصوتي — المعلّم اللحظيّ

Vapi منصّة ذكاء اصطناعيّ صوتيّة. تُنسّق الأجزاء الثلاثة المتحرّكة لأيّ محادثة منطوقة كي لا تضطرّ لبنائها بنفسك:

  1. تحويل الكلام إلى نص (transcriber) — هنا Deepgram (نموذج nova-3) يُحوّل صوت ميكروفون المستخدم إلى نصّ.
  2. «العقل» (نموذج اللغة LLM) — OpenAI GPT-4 يستقبل ذلك النصّ مع تعليمة نظام للمعلّم فيُنتج ردًّا.
  3. تحويل النص إلى كلام (voice) — ElevenLabs (11labs) ينطق الردّ بالصوت المختار.

والأهمّ أنّ Vapi في هذا التطبيق يعمل كليًّا داخل المتصفّح عبر @vapi-ai/web. لا دخل للخادم في المكالمة الحيّة.

إعداد المساعد

الدالّة configureAssistant(voice, style) في lib/utils.ts تبني كائن إعداد Vapi. لاحظ العلامات النائبة {{topic}} و{{subject}} و{{style}} — تُملأ لحظة بدء المكالمة (انظر أدناه):

lib/utils.ts — configureAssistant() (مقتطف)
const voiceId = voices[voice][style] || "sarah";   // اختيار صوت ElevenLabs
return {
  transcriber: { provider: "deepgram", model: "nova-3", language: "en" },
  voice:       { provider: "11labs", voiceId, stability: 0.4, /* … */ },
  model: {
    provider: "openai", model: "gpt-4",
    messages: [{ role: "system", content: `You are a tutor teaching {{ subject }}…
      Stick to the topic {{ topic }}. Keep your style {{ style }}. …` }],
  },
};

تشغيل المكالمة: components/CompanionComponent.tsx

مكوّن العميل هذا هو واجهة الجلسة الحيّة بالكامل. يقوم بـ (١) بدء/إيقاف المكالمة، و(٢) الاشتراك في أحداث Vapi، و(٣) عرض النصّ والرسوم المتحرّكة. وتتحكّم آلة حالات محليّة CallStatus (INACTIVE → CONNECTING → ACTIVE → FINISHED) بالزرّ والمظهر.

components/CompanionComponent.tsx — بدء المكالمة
const handleCall = async () => {
  setCallStatus(CallStatus.CONNECTING);
  const assistantOverrides = {
    variableValues: { subject, topic, style },  // تملأ {{subject}} إلخ.
    clientMessages: ["transcript"],           // إرسال النصوص إلى المتصفّح
    serverMessages: [],                          // لا webhooks على الخادم
  };
  vapi.start(configureAssistant(voice, style), assistantOverrides);
};
components/CompanionComponent.tsx — الاستماع لأحداث Vapi
vapi.on('call-start', () => setCallStatus(ACTIVE));
vapi.on('call-end',   () => { setCallStatus(FINISHED); addToSessionHistory(companionId); });
vapi.on('message',    (m) => {          // النصّ المباشر
  if (m.type === 'transcript' && m.transcriptType === 'final')
    setMessages(prev => [{ role: m.role, content: m.transcript }, ...prev]);
});
vapi.on('speech-start', () => setIsSpeaking(true));   // تشغيل رسم الموجة الصوتيّة
vapi.on('speech-end',   () => setIsSpeaking(false));
ملاحظة

المكالمة الحيّة تحتاج متغيّر البيئة في المتصفّح NEXT_PUBLIC_VAPI_WEB_TOKEN (انظر lib/vapi.sdk.ts). إن لم يُضبَط، تُعرَض الصفحة لكن vapi.start() لن يستطيع الاتّصال. وهذا منفصل تمامًا عن Clerk/Supabase.

كيف تتواصل Clerk وVapi وSupabase

هي لا تتّصل في مثلّث — بل تتّصل عبر التطبيق. إليك الصورة نفسها من زاوية المسؤوليّات و«الغراء» الذي يربط كلّ زوج:

flowchart TB
  APP(["Converso app
(Next.js + Browser)"])
  CLERK["🔐 Clerk"]
  SUPA["🗄️ Supabase"]
  VAPI["🎙️ Vapi"]
  APP -- "auth() · cookies · has()" --- CLERK
  APP -- "supabase.from(...)
server actions" --- SUPA
  APP -- "vapi.start/stop · events" --- VAPI
  CLERK == "issues JWT that
Supabase verifies (Third-Party Auth)" ==> SUPA
الرابط المباشر الوحيد بين خدمتين هو Clerk → Supabase، وهو غير مباشر: يُصدِر Clerk رمزًا، وSupabase مُهيّأ للثقة به.

Clerk ↔ Supabase (الهويّة تفتح البيانات)

يُصدِر Clerk رمز جلسة JWT؛ وتُرفِقه createSupabaseClient() بكلّ استعلام؛ ويتحقّق منه Supabase عبر نطاق Clerk المُسجَّل تحت مصادقة الطرف الثالث، ثم تُصفّي RLS الصفوف حسب sub. لا webhooks — بل تحقّق من الرمز وقت الطلب.

التطبيق ↔ Vapi (المتصفّح يقود المكالمة)

يفتح المتصفّح مكالمة WebRTC إلى Vapi ويستمع للأحداث. Vapi لا يمسّ Supabase ولا Clerk أبدًا. وعند انتهاء المكالمة، يستدعي المتصفّح إجراء خادم لحفظ الجلسة.

التطبيق ↔ Clerk (الهويّة + الفوترة)

clerkMiddleware() يتحقّق من الملفّ (cookie) في كلّ طلب؛ وauth()/currentUser() يقرآن المستخدم؛ وhas() يفحص خطّة الاشتراك.

بخصوص الـ webhooks

سألتَ عن الـ webhooks. هذا التطبيق لا يستخدم أيًّا منها. فـ Vapi مُهيّأ بـ serverMessages: [] (بلا نداءات خادم)، وفحوصات خطط Clerk تحدث وقت الطلب عبر has(). كلّ شيء إمّا إجراء خادم (المتصفّح → خادم Next → Supabase) أو تحقّق من رمز وقت الطلب. يمكن إضافة الـ webhooks لاحقًا (مثلًا serverMessages في Vapi لتسجيل النصوص كاملة على الخادم، أو webhooks من Clerk لمزامنة المستخدمين) — لكنّها ليست جزءًا من التصميم الحاليّ.

07

Vapi ↔ Supabase: كيف تتدفّق البيانات

لا يتحدّث Vapi وSupabase مباشرةً أبدًا. المتصفّح هو الجسر. أثناء المكالمة، تعيش النصوص في حالة React فقط (تُعرَض ولا تُحفَظ). وعند انتهاء المكالمة، يُطلِق المتصفّح إجراء خادم يكتب صفًّا واحدًا في session_history.

sequenceDiagram
  actor U as User
  participant CC as CompanionComponent (browser)
  participant V as Vapi (cloud)
  participant SA as addToSessionHistory (server action)
  participant S as Supabase
  U->>CC: Click "Start Session"
  CC->>V: vapi.start(assistant, { variableValues })
  V-->>CC: 'call-start' → status ACTIVE
  loop during the lesson
    U->>V: speaks (mic audio)
    V-->>CC: 'message' transcript → append to React state
    V-->>CC: 'speech-start' / 'speech-end' → animate soundwave
  end
  U->>CC: Click "End Session"
  CC->>V: vapi.stop()
  V-->>CC: 'call-end'
  CC->>SA: addToSessionHistory(companionId)
  SA->>SA: userId = (await auth()).userId
  SA->>S: insert session_history { companion_id, user_id }
  S-->>SA: ok (RLS: sub === user_id)
البيانات الوحيدة من Vapi التي تصل إلى Supabase هي: «هذا المستخدم أكمل جلسة مع هذا الرفيق».
lib/actions/companion.actions.ts — addToSessionHistory()
export const addToSessionHistory = async (companionId) => {
  const { userId } = await auth();               // معرّف Clerk
  const supabase = createSupabaseClient();        // يحمل رمز Clerk JWT
  await supabase.from('session_history').insert({
    companion_id: companionId,
    user_id: userId,
  });
};

هذا السجلّ الواحد يُغذّي «الدروس المكتملة» و«الجلسات الأخيرة» و«الجلسات المكتملة مؤخّرًا» في الصفحة الرئيسيّة — وكلّها تُقرأ عبر getRecentSessions() / getUserSessions()، اللتين تستخدمان تضمينًا عبر مفتاح Postgres الأجنبيّ لجلب الرفيق الكامل لكلّ جلسة:

getUserSessions() — القراءة مع التضمين
supabase
  .from('session_history')
  .select(`companions:companion_id (*)`)  // تتبّع المفتاح الأجنبي → الرفيق الكامل
  .eq('user_id', userId)
  .order('created_at', { ascending: false });
لماذا كان المفتاح الأجنبيّ مهمًّا

ذلك التضمين companions:companion_id (*) يعمل فقط لأنّ لدى session_history.companion_id مفتاحًا أجنبيًّا يشير إلى companions(id). والأمر نفسه ينطبق على bookmarks. غياب هذا المفتاح هو سبب تعطّل /my-journey حتى أُنشئت الجداول بالشكل الصحيح.

مثال ب — تشغيل درس صوتيّ

مفاهيم أساسية والاعتماديّات

مكوّنات الخادم مقابل مكوّنات العميل

افتراضيًّا، ملفّات App Router هي مكوّنات خادم (Server Components) — تعمل على الخادم، ويمكن أن تكون async، وتصل إلى Supabase/Clerk مباشرةً (مثاليّة لصفحات جلب البيانات). أمّا الملفّ الذي يحتاج واجهات المتصفّح أو الحالة أو معالِجات الأحداث فيبدأ بـ 'use client' (مثل CompanionComponent وCompanionForm وSearchInput). ولا بدّ أن يكون Vapi مكوّن عميل لأنّه يستخدم الميكروفون.

إجراءات الخادم (Server Actions)

الملفّات المُعلَّمة بـ 'use server' تُتيح دوالّ يستدعيها العميل كأنّها await عاديّة، لكنّها تُنفَّذ على الخادم. هكذا يُشغّل المتصفّح كتابات قاعدة البيانات بأمان (مثل addToSessionHistory وcreateCompanion) دون كشف بيانات اعتماد قاعدة البيانات.

أمان مستوى الصف (RLS)

التفويض يعيش داخل Postgres، لا في شيفرة التطبيق فقط. حتى لو نسي استعلامٌ إضافة .eq('user_id', …)، فستظلّ RLS تقصر الصفوف على المستخدم الحاليّ. وترتكز السياسات على مطالبة Clerk sub.

مرجع متغيّرات البيئة

المتغيّرات التي تبدأ بـ NEXT_PUBLIC_ مكشوفة للمتصفّح (مفاتيح عموميّة آمنة). وكلّ ما عداها للخادم فقط (سرّيّ).

المتغيّريستخدمهأين
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEYعميل Clerk.env
CLERK_SECRET_KEYخادم Clerk (سرّيّ).env
NEXT_PUBLIC_CLERK_SIGN_IN_URL / …SIGN_UP_URLإعداد مسارات Clerk.env
NEXT_PUBLIC_SUPABASE_URLرابط مشروع Supabase.env.local
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEYمفتاح واجهة Supabase.env.local
NEXT_PUBLIC_VAPI_WEB_TOKENعميل Vapi في المتصفّح.env (مطلوب للصوت)
خطوة في لوحة التحكّم ليست في الشيفرة

كي يثق Supabase برموز Clerk، يُسجَّل Clerk كـ مزوّد مصادقة طرف ثالث في لوحة Supabase (باستخدام نطاق Clerk، مثل sure-piranha-43.clerk.accounts.dev)، ويُفعَّل تكامل Supabase في لوحة Clerk. هذا الإعداد هو ما يجعل سلسلة Clerk→Supabase كلّها تعمل — ولا يظهر منه شيء في المستودع.
